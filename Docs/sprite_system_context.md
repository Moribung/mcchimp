# McChimp Sprite System — Claude Code Context Document

> Drop this file into every Claude Code session working on the sprite system.
> It defines the full asset spec, file contract, compositor API, and scene config schema.

---

## What this system is

A programmatically generated pixel art sprite library plus a SvelteKit canvas compositor
for the MMA Career Trivia game on mcchimp.com. All sprites are generated by Python scripts
using Pillow — no external art assets, no Supabase Storage, no CDN. Output PNGs live in
`static/sprites/` and are served as static files by Cloudflare Pages.

The compositor is a single Svelte 5 component (`SceneCompositor.svelte`) that accepts a
scene config object, loads the relevant sprites, and composites them onto a canvas in z-order.

## Sprite specification

### Canvas / resolution

- **Native sprite size: 32 × 48 px** for full-height figures (fighters, referee)
- **Scene canvas: 96 × 48 px** — wide enough for two fighters + referee side by side
- All sprites use a transparent background (RGBA PNG) except backgrounds
- Background sprites are exactly **96 × 48 px**, opaque
- Display scale: 8× via CSS (`image-rendering: pixelated`) → renders at 768 × 384 on screen

### Style reference

The style is chunky, readable pixel art — body parts are rectangular blocks, not curves.
Each body region is one or two filled rectangles. No anti-aliasing, no sub-pixel detail.
A head is a 6×6 block. An arm is a 3×4 block. Legs are 3×6 blocks.
This is the style established in the working interactive demo already built in this project.

## Asset inventory

### Backgrounds (96 × 48 px, opaque)

| ID                | Filename                  | Description                            |
|-------------------|---------------------------|----------------------------------------|
| `bg_empty`        | `bg_empty.png`            | Dark arena, no crowd. Spotlight only.  |
| `bg_crowd_medium` | `bg_crowd_medium.png`     | ~40% crowd fill, scattered dot heads.  |
| `bg_crowd_packed` | `bg_crowd_packed.png`     | Dense crowd, full upper half.          |

All three share the same structure:
- y 0–2: near-black ceiling `(9, 10, 17, 255)`
- y 3–28: crowd zone, dark `(12, 14, 22, 255)` base
- y 29: horizontal rail divider `(25, 28, 42, 255)`
- y 30–35: lower arena wall `(18, 21, 33, 255)`
- y 36–47: mat floor `(23, 31, 55, 255)` with a lighter line at y=36

Crowd dots are single pixels drawn at random positions in y 3–27 using CROWD_1–5.
Empty: 0 dots. Medium: ~60 dots. Packed: ~160 dots.

### Octagon overlays (96 × 48 px, transparent)

Drawn over the background as mat markings only — no solid fill, transparent base.

| ID           | Filename         | Description                                      |
|--------------|------------------|--------------------------------------------------|
| `oct_ufc`    | `oct_ufc.png`    | White octagon outline on mat floor area.         |
| `oct_cage`   | `oct_cage.png`   | Sparse chain-link dot pattern + two cage posts.  |
| `oct_ring`   | `oct_ring.png`   | Boxing ring: two corner posts, two rope lines.   |

All markings are drawn only in the mat zone (y 36–47) and lower wall (y 30–35).
Line colour: `(255, 255, 255, 60)` — semi-transparent white, subtle.

### Fighter sprites (32 × 48 px, transparent)

The fighter always faces **left** in the sprite file.
For the opponent, the compositor mirrors horizontally using `ctx.scale(-1,1)` —
no second sprite file is needed.

At 32×48 the fighter body uses these approximate region blocks:

```
Head:      x=13, y=7,  w=6, h=6   (skin)
Hair:      x=13, y=5,  w=6, h=2   (hair) + side pixels
Eyes:      x=15, y=9,  w=1, h=1   and x=17, y=9 (eye)
Torso:     x=11, y=13, w=10,h=9   (skin)
Shorts:    x=11, y=22, w=10,h=5   (shorts colour)
Leg L:     x=12, y=27, w=3, h=7   (skin)
Leg R:     x=17, y=27, w=3, h=7   (skin)
Feet:      x=12, y=34, w=3, h=1   and x=17,y=34 (outline)
Arm L up:  x=8,  y=13, w=3, h=4   (skin) + glove x=8,y=9,w=3,h=3
Arm R up:  x=21, y=13, w=3, h=4   (skin) + glove x=21,y=9,w=3,h=3
```

These are the exact coordinates used in the working interactive demo.
All new fighter sprites must use these same anchor positions for head, torso, and feet
so that the compositor can place all poses consistently.

| ID                 | Filename                   | Pose description                              |
|--------------------|----------------------------|-----------------------------------------------|
| `fighter_side`     | `fighter_side.png`         | Side-on stance facing left, guard up          |
| `fighter_forward`  | `fighter_forward.png`      | Forward-facing neutral stance                 |

## sprites.json manifest schema

```json
{
  "version": "1.0",
  "sceneWidth": 96,
  "sceneHeight": 48,
  "displayScale": 8,
  "sprites": [
    {
      "id": "fighter_forward",
      "file": "fighter_forward.png",
      "width": 32,
      "height": 48,
      "anchorX": 16,
      "anchorY": 47
    },
    {
      "id": "ref_neutral",
      "file": "ref_neutral.png",
      "width": 24,
      "height": 40,
      "anchorX": 12,
      "anchorY": 39
    }
  ]
}
```

`anchorX`/`anchorY` are the feet pixel within the sprite. When the compositor places
a sprite at scene position `(sceneX, sceneY)` it draws the sprite so that its anchor
pixel lands on that coordinate: `drawX = sceneX - anchorX`, `drawY = sceneY - anchorY`.

## Scene config object

```js
{
  background: 'bg_crowd_packed',
  octagon:    'oct_ufc',
  player: {
    sprite: 'fighter_side',
    sceneX: 24,
    sceneY: 44,
    flip:   false
  },
  opponent: {
    sprite: 'fighter_side',
    sceneX: 72,
    sceneY: 44,
    flip:   true
  },
  referee: {
    visible: true,
    sprite:  'ref_raise',
    sceneX:  88,
    sceneY:  43
  }
}
```

The MMA game builds this config from career state. Examples:

- KO win: `player.sprite = 'fighter_win_one'`, `opponent.sprite = 'fighter_loss'`, `referee.visible = false`
- Decision win: `player.sprite = 'fighter_win_both'`, `referee.sprite = 'ref_raise'`, `referee.visible = true`
- Loss: `player.sprite = 'fighter_loss'`, `opponent.sprite = 'fighter_win_one'`
- Prefight: both `fighter_side`, referee hidden

## SceneCompositor.svelte API

```svelte
<SceneCompositor config={sceneConfig} />
```

Props:
- `config` — scene config object as above

The component uses Svelte 5 runes (`$props`, `$effect`). It:
1. Loads `sprites.json` once on mount via fetch('/sprites/sprites.json')
2. Preloads all PNG files via `new Image()` into a Map keyed by sprite ID
3. On `config` change: clears canvas, draws in this z-order:
   - background
   - octagon overlay
   - opponent (with flip if set)
   - player
   - referee (if visible)
4. Flip is implemented as: `ctx.save(); ctx.translate(drawX*2 + w, 0); ctx.scale(-1,1);`
   then draw at 0,drawY; then `ctx.restore()`
5. Canvas element has `style="image-rendering: pixelated; width: 768px; height: 384px"`
   (the 8× display scale of the 96×48 native canvas)

## Python generation conventions

```python
from PIL import Image, ImageDraw
from palette import *

W, H = 32, 48
img = Image.new('RGBA', (W, H), (0, 0, 0, 0))
d = ImageDraw.Draw(img)

def rect(x, y, w, h, c):
    d.rectangle([x, y, x+w-1, y+h-1], fill=c)

# drawing calls using rect() only
# coordinates are native pixels (pre-scale)

img.save('../../static/sprites/fighter_forward.png')
```

Backgrounds use `(0, 0, 0, 255)` as base fill. All others transparent.
Use only `rect()` — no lines, no ellipses, no anti-aliased drawing.
Keep body parts as rectangular blocks matching the style reference above.
